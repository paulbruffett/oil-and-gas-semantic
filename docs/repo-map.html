DESIGN.md is the source of truth — this page is the map into the code that realizes it.

A vendor-neutral reference for comparing data platforms and coding agents.

One deterministic substrate — an OSDU-grounded oil & gas production dataset, governed KPIs, a knowledge graph, and gold answers — built once so two very different comparisons can be run fairly on a common footing.

AXIS A · platform

Which data + semantic stack?

Instantiate the base on Fabric / Snowflake / Databricks. Assessed by demonstrating the pattern end-to-end — the winning contest build becomes that platform's reference.

AXIS B · assistant

Which coding assistant?

Each assistant builds a parallel implementation from the same spec, in its own fork from a frozen tag. Assessed by an objectively-anchored code-quality review.

The axes are orthogonal: the shared substrate and deterministic data let you swap the platform or the assistant independently and still compare apples to apples.

The stratigraphic column

Six platform-neutral layers

The architecture is specified as six neutral layers ADR 0003; every platform or assistant maps them onto its own technology. Read them as a core sample — surface data at the top, governance cutting across the whole column. The src/ packages implement the shell of layers 1–5; layer 6 is cross-cutting policy.

Where you can substitute one thing for another

The three seams

A seam is a boundary where an implementation can be swapped without touching callers. The whole design turns on three of them — tests drive the system through these, and contestants honor them rather than the internals.

Data seam

Generator output

The deterministic generator is the single substrate; gold answers are computed here from the same in-memory columns the Parquet is written from, so the two can never drift. The highest engineering-test seam.

generate_dataset() · gold.py
Semantic seam

OSI metric definitions

Metrics live once and compile down per platform. Agents select a governed metric rather than authoring SQL/DAX — so a correct selection yields a correct query by construction.

semantic/metrics.yaml · compile.py
Answer seam

Answer-submission schema

The one thing every agent must honor: NL answer + key values + optional provenance + a behavior flag. It enables grading without constraining agent design.

answer_submission.schema.json

src/ · layer 1–2 · the data seam

oag_generator

A deterministic OSDU/PDM-shaped generator. One seed drives every draw; it emits Parquet tables, a secondary OSDU JSON view, and the six themes' gold answers — all stamped with a content hash so any two instantiations are provably comparable.

Determinism is engineered, not hoped for

One default_rng(seed) drives the whole build. The optional water/gas breakthrough scenario draws from a separately-derived stream so turning it on can't perturb the main sequence — and worst-actor pins always consume the draws they replace, keeping every other well byte-identical.

The forecast is a decline curve; "expected" is what it emits

Per-well Arps hyperbolic parameters are drawn, and the full-uptime potential becomes the Forecast series. Actual oil applies a performance factor and the on-stream fraction. That forecast is the baseline for every variance, deferment, and decline KPI ADR 0006.

Every dataset is stamped with a content hash

The first 12 hex of a sha256 over the sorted, resolved config. The same hash stamps dataset.json and every OSDU manifest, so the canonical and secondary views can never diverge. The frozen contest dataset is 12a110eecfe2.

Gold is computed beside the data — the hero flag

Surveillance sums actual vs forecast oil over the trailing window and flags wells materially below a threshold ADR 0009. Because it reads the same rounded columns the Parquet holds, an independent recompute from Parquet reproduces it exactly — which is precisely what the tests assert.

src/ · layers 3–5 · semantic + knowledge + answer seams

oag_semantic

The hero reference implementation of the production-surveillance question, end-to-end. Its own docstring maps its modules to the three seams: semantic (OSI → SQL via DuckDB), knowledge (the LPG), and answer (the submission schema + grader).

Semantic seam — the query shape is the question, only the tokens come from the manifest

The compile pulls governed measures by name from the SemanticLayer, registers each model's Parquet under its manifest table alias, and builds SQL whose every identifier — table, join key, column expr — is manifest-sourced. It mirrors gold.py exactly so the two cannot diverge ADR 0011.

Knowledge seam — mapping a business phrase to a governed metric

The LPG does entity resolution (field names + synonyms), the well→field rollup, and term resolution — "watering out" → water_cut, "below expected" → production_efficiency. A substring fallback lets a term embedded in a longer question still resolve.

The baseline agent selects a metric deterministically — no LLM

The SemanticBaselineAgent is deterministic-in, deterministic-out and tolerance-gradable. It resolves the phrase to the governed metric through the LPG (step 1), optionally scopes to a field (step 2), then computes via the DuckDB compile (step 3) and emits provenance. An agentic variant using an LLM tool-loop is the open design space contestants explore ADR 0005.

Answer seam — grading is exact-set plus tolerance

The flagged set must match gold exactly; the per-well values must match within a small relative tolerance (DuckDB sums vs Python sums agree to float, not bit, precision). This is rubric dimension 1 — objective, never voted.

Governed metrics are authored once in semantic/metrics.yaml as OSI/MetricFlow docs — production_efficiency is a ratio of actual_oil / expected_oil; water_cut, gor, uptime_pct are derived. KPIs that aren't a single aggregate (deferred volume, annualized decline, days-down) are assembled by the matching compile.compute_*, which reproduces gold.

Functional correctness = behavior AND values, unanswered = failed

Every gradable question is walked. A submission is schema-validated at grading time (invalid → structural failure, not a crash). A question is correct only if its behavior matches (straight = answered; ambiguous = clarification-requested; trap = refused-data-quality) and its values match. An unanswered gradable question fails — you can't inflate the pass rate by answering only the sure ones ADR 0025.

Sealed custody — provably not tailored to observed outputs

Round-2 change requests and adversarial paraphrases are authored before the fork tag and held outside version control; only their digest is committed pre-tag. The digest is content-addressed with no mtime/uid — re-derivable by hand — so oag-seal verify at release proves the set wasn't edited to match what contestants produced ADR 0028.

spec/ · the agent-layer contract

The question catalog & the answer seam

spec/questions/catalog.yaml is a standalone, versioned artifact: the six use-case themes plus a nine-question adversarial tier. Each question's gold_id is the join key to a co-generated gold answer, and its grading block declares the exact shape the harness grades — so the spec and the grader cannot diverge.

The six themes — hero ★ built end-to-end, 2–6 gold co-generated

1HERO ★ Production surveillance · "which wells produced below expected oil rate this week, and by how much?"variance · efficiency
2Deferment & downtime attribution · "what did we defer last month, and top causes?"deferred vol · uptime%
3Decline & trend · "12-month oil decline; flag wells declining faster than forecast."decline rate · cumulative
4Well-test & allocation validation · "stale tests or anomalous allocation?"days-since-test · alloc var

The adversarial tier — reasoning graded as objectively as arithmetic

Nine questions spanning the themes rather than forming a seventh ADR 0024. The gold encodes the expected behavior, so "should have asked" and "should have refused" grade as cleanly as a numeric answer. A deterministically-seeded worst-actor well anchors every compound intersection, guaranteeing non-empty gold on any seed.

Compound ×3
expects: answered

Span ≥2 governed metrics; gold = the intersection of two straight golds (e.g. below-expected and stale test).

Ambiguous ×3
expects: clarification-requested

Under-specified ("which wells are underperforming?"). The right move is to ask, not guess. Behavior-only.

Trap ×3
expects: refused-data-quality

Probe the seeded trap well NO 15/9-F-1, whose only well test predates the data. The responsible answer is a refusal.

Why a held-out seed?

The gold shipped in the fork is build-time collateral, not the graded number. regenerate_at_seed overrides only the seed and holds the substrate fixed — a fair fresh draw of the same problem. Implementations must be seed-agnostic ADR 0016.

Why locus adherence?

Each change request declares the seam it should land at. locus.py splits the diff into in-locus vs out-of-locus; raw line counts are reported, not scored. Maintainability is measured behaviorally, not just by panel opinion ADR 0015.

DESIGN.md §8 · our own TDD, not the product

Engineering tests

Two kinds of "test" are kept strictly separate. The product is the Axis-A/B assessment above. The engineering tests — 22 files, ~250 pytest cases — verify our base-collateral code, driving each module through its highest public seam against fixed-seed fixtures and asserting on observable output, never private state.

Generator

test_generator.py (940 lines) recomputes each KPI from raw Parquet and asserts gold matches; test_conformance checks the OSDU profile; test_contest_config sweeps held-out seeds.

Semantic

test_reference_compile.py reproduces gold and eval()s the metric formula string against DuckDB; "poison" rows prove the filters hold; test_agent asserts determinism.

Harness

test_harness_functional asserts an oracle scores 6+9 at 100%; test_harness_evalseed proves fork-time answers fail on the held-out seed; custody, panel, webapp all covered.
