# Axis-B discrimination scope: sealed change-request round, neutral webapp vertical, adversarial question tier — every addition carries an objective anchor

**Context.** With the data-seam split (ADR 0012), the per-theme contest work (LPG + agent wiring + platform mapping from a visible hero skeleton) is within easy reach of every frontier model, risking a ceiling effect where the assessor panel ends up scoring style rather than capability.

**Decision.** Add three contest-scope elements, each with an **objective anchor** so panel scoring stays a tiebreaker, not the measurement: (1) a **sealed change-request round** — after initial builds, every contestant applies the identical, until-then-private change set (KPI redefinition, schema migration, failing-gold bug report) to its own fork; graded on re-run correctness plus diff blast radius, making maintainability behavioral; (2) an **operations-console webapp** as a contest vertical, specified in the shell as **platform-neutral functional requirements plus a data contract** (screens per use-case theme, interactions, "displayed values match gold for the frozen seed") — never a tech stack, which is itself a graded choice; (3) an **adversarial question tier** in the shell question catalog — compound, ambiguous (assumptions/clarification expected), and trap questions (correct answer is a data-quality refusal) with the expected behavior gold-encoded.

**Why.** These attack the ceiling effect where models actually diverge — change absorption on their own code, sustained frontend build quality, and reasoning under ambiguity — while keeping functional correctness objectively computable and the base collateral platform-neutral (ADR 0003).
