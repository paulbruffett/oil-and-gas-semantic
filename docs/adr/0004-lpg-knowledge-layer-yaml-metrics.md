# Knowledge layer as a Labeled Property Graph (not RDF/OWL); metrics as neutral YAML

**Context.** The project name foregrounds "semantic/ontology," so a future reader will expect RDF/OWL. But the three target platforms converge elsewhere: **Snowflake** (Semantic Views, YAML; co-author of the Open Semantic Interchange standard), **Azure Fabric** (Power BI/DAX semantic models + **Fabric Graph = Labeled Property Graph with GQL**, explicitly *no* RDF), and **Databricks** (Unity Catalog Metric Views; OWL reasoning only via the add-on OntoBricks). None run native RDF/OWL reasoning. The six production-operations use cases are **quantitative aggregations** that need **semantic mapping** (e.g. "watering out" → watercut) and **relationship navigation** (well → facility → field → operator), not description-logic inference.

**Decision.** Represent **layer 3** as a neutral **YAML metric spec** and **layer 4 (knowledge)** as a neutral **Labeled Property Graph** (entities, typed relationships, hierarchy, business vocabulary/synonyms) — **not** full RDF/OWL. A full **RDF/OWL ontology with reasoning** (SPARQL/SHACL/SWRL) is retained as an **optional advanced track** for exercising dedicated reasoning tooling (OntoBricks / Fabric IQ / a standalone triplestore) and as a deliberate comparison dimension.

**Why.** LPG + YAML metrics map to the platforms' native capabilities, give agents the grounding this workload actually uses, and align with the Open Semantic Interchange direction — without paying the OWL authoring cost for inference features this workload won't exercise. OWL stays available precisely where reasoning tooling is the explicit thing under test.
