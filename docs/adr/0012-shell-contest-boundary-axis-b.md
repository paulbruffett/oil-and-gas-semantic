# Shell/contest boundary: hero built in-repo as scaffolding; use cases 2–6 split at the data seam; Axis-B work built in forks from a frozen tag

**Context.** ADR 0003 scopes the base collateral to specs + one runnable artifact (the generator), but the slice backlog had this repo implementing all six use cases end-to-end — leaving no distinctive work for the Axis-B coding-assistant comparison, with contest mechanics (fork point, what's given vs. built) undefined. Gold answers, however, must always come from the base — contestants cannot author their own grading key.

**Decision.** The **hero slice (#3) is built end-to-end in this repo** as validated scaffolding (a deliberate, bounded deviation from ADR 0003's "specs-only": it proves the gold/OSI/answer seams and gives every contestant the same skeleton, but is **not graded**). For use-case themes 2–6 the shell keeps only the **data-seam half** (generator entities, canonical tables, gold answers, question text, OSI metric definitions + MetricFlow validation); the **LPG/agent/platform-wiring half** is contest work, specified in open `axis-b-contest` issues and built by each assistant **in its own fork** from a **tagged fork point** cut after all shell issues + Volve calibration (#13) land, with the dataset config hash frozen. The in-repo build is neutral scaffolding, excluded from Axis-B scoring; Claude Code competes later from the same fork point as every other assistant, with effort metering on.

**Why.** This keeps the objective grading dimension sound (gold is shell-authored), keeps the comparison meaningful (each assistant does the same substantial, unseen work from the same frozen substrate), and de-risks the specs (the hero proves the whole path before anyone competes).
